- 对于大型工业边缘网关，核心原则：主程序不应该知道任何底层协议库的细节。因此，对于不同的物联网协议，必须抽象出相同的接口。 引入协议的适配器 (Adapter) 层，构建「主网关程序-协议适配器-协议」这样的层次关系。协议适配器层 (Adapter Layer) 的责任： 资源和网络管理者，负责读取配置、拨号、建立连接、保持心跳 (Keep-alive) 以及在网络波动时进行重连。
- 配置预编译。
- 数据转换层：网关主程序通过统一的“数据转换层”或“物模型映射层”将它们反序列化为最终的业务数值。
- 每个采集插件的Initialize 初始化，只应做两件事： 1. 读取点位配置文件  2. 注册设备到后台。读完配置就应该注册

## 编程随想

golang: 当希望行为相同的不同对象时，我们返回接口。这是工厂的设计模式，返回不同对象的实例，掩盖具体实现。
接受接口，返回结构体；接口属于使用者。这个项目中，协议接口属于Adapter, 因为它才是这些协议的真正用户。

1. 谁接受接口，当然接口的用户，谁使用，谁接口。谁返回接口体，当然是是实现接口的，被抽象的。
2. 接口属于使用者，但不表示接口的代码与接口用户代码放在一起，接口的代码应当与被抽象的代码放在一起。



设备读取：

```text
HandleReadCommand
    │
    ▼
ProtocolDriverRegistry     ← 按协议类型查找驱动
    │
    ▼
ProtocolDriver (e.g. ModbusDriver)
    │
    ▼
DeviceProfile.ReadStrategy  ← 每个设备挂一个"读策略"对象（结构体 ）
    │                        （批大小、寄存器排列、重试逻辑等）
    ▼
BatchExecutor     ← 按策略切片，逐批执行实际读取
```

- 优点： 驱动协议干净，策略可独立变化，新增设备只需配置不需改代码。
- 缺点： 策略对象字段可能膨胀，复杂设备的策略难以用纯配置表达。



程序设计的思路：

1. 物联网协议层必须是独立模块，隔离协议差异。仅仅对外暴露少数接口供采集器使用。 技术上用golang接口，多态，所有协议统一入口。
2. 资源配置采集edge-sdk规范，toml格式。excel资源配置表转toml放在预编译阶段。
   3. 设备配置驱动采集行为，而不是硬编码
```text
           设备配置层（告诉驱动如何读取设备） 
                    
转换层     执行层（BatchExecutor，执行采集的动作 ）     
            驱动层 （驱动不认识设备名，只认设备配置对象）
            协议层（定义通用读写能力：不认识设备，只认读法）
```

1. 执行层从设备配置层获取设备列表和读策略，调用对应的协议驱动层读写方法。
2. 协议驱动层，基于具体的协议实现，读写并且返回数据给执行层。
3. 执行层，将得到的数据列表，传给转换层
4. 转换层，设备配置层，进行转换，得到最终数据返回给执行层。
5. 执行层再将最终数据异步给持久化层。


转换只会规则是设备级别的，因此必须从配置中来，输出标准化的数据结构。

数据格式的设计：

设备表（描述每个设备）：
- 设备唯一标识（全局唯一）
- 名称
- 描述
- 设备使用的协议
- 采集周期
- 每次请求最大的资源数量
- supports_bulk_read = false（）
- hide: 是否启用

设备表由三部分构成：  基本设备属性（id，名称采集） ReadStrategy策略对象字段（采集周期，最大资源数量，batch_strage "(8000, 111)"）

设备资源表 （定义设备的所有可读写的资源）
- 资源ID （不要求全局唯一，设备内唯一即可）
- 资源名称
- 描述
- 数据类型
- 默认值
- 范围 （用于校验数据是否有效）   
- hide (该资源保留，暂时不处理)
- 解析函数。

设备资源表由三部分构成： 基本资源属性（名称，id，类型等 ），，转换对象字段（范围，默认值，转换方式即解析函数）


旧的IBIMS项目同时采用Excel 和 json配置文件，完全绕过了edge-sdk需要的toml，后续的设备读写同样绕过了sdk。 我认为最好的做法是：**恪守项目标准，仅仅使用toml**。把Excel 转换toml的步骤作为独立脚本，放在`预编译配置 (Pre-compiled Configuration)`的阶段，即在程序启动前完成转换（该独立脚本作为构建/部署流水线 Makefile执行的一部分），这更符合开源社区和现代软件工程（尤其是云原生和 DevOps 领域）的主流实践。我的理由：

1. 既然采用了edge-sdk框架，应该遵守其设计标准，让sdk自动加载配置，而不是增加应用代码的复杂度。
2. 职责分离， 保持主程序的纯粹性，不应该读excel，不应该把转换任务放在运行时。
3. 依赖最小化原则。解析 Excel（如 .xlsx）需要引入体积较大的第三方库（例如 Python 的 pandas/openpyxl，Go 的 excelize）。如果放在主程序中，会无故增加主程序的二进制体积和内存占用, 性能和安全性。
4. 故障前置，如果Excel格式人工维护的格式，极其容易出错，生产环境一旦出错，可能需要重新编译主程序。保只有格式绝对正确、验证通过的 TOML 文件才会喂给主程序。即便Excel生成的格式不正确，也可以手动调整toml文件。
5. 配置文件的可测试性，生成toml更容易单元测试和版本之前跟踪变化。

Bootstrap() 内部执行流程
用户必须实现`ProtocolDriver` 接口。 调用 Bootstrap() 后，SDK 会自动完成以下全部工作，开发者无需手动处理：
```text
Bootstrap() 调用
│
├── 1. 解析命令行参数（-p, -cd, -cf, -i, -r, -cp 等）
├── 2. 加载配置文件（configuration.yaml）或从 Consul 拉取配置
├── 3. 向 Registry（Consul）注册本服务
├── 4. 连接 EdgeX 核心服务
│        ├── core-metadata（设备、Profile、ProvisionWatcher）
│        └── core-data / Message Bus（事件上报）
├── 5. 初始化缓存（设备、Profile、ProvisionWatcher 等）
├── 6. 调用 driver.Initialize(sdk) ← 用户自定义初始化逻辑
├── 7. 启动 REST API HTTP Server（设备命令路由）
├── 8. 启动 AutoEvents（自动采集事件）
├── 9. 调用 driver.Start() ← 用户初始化后置逻辑（新版本）
└── 10. 阻塞等待关闭信号（SIGTERM/SIGINT），触发 driver.Stop()
```
